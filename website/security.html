Security

Updated: 2026-04-20

This page describes CallSight's current security controls. It is factual about what we do today; it does not claim certifications we do not yet hold.

Transport security

  • All API, webhook, and WebSocket traffic is TLS 1.2+ in production.
  • HSTS enforced at the edge; the FastAPI app emits CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, and Permissions-Policy on every response.
  • Outbound webhook dispatch blocks private and link-local IP ranges (RFC 1918, 169.254.0.0/16, loopback, ULA) and pins the resolved IP at connect time to defend against DNS rebinding.
  • Database connections validate the CA chain and hostname; certificate verification is never disabled outside DEBUG mode.

Data at rest

  • OAuth access and refresh tokens are Fernet-encrypted (AES-128-CBC + HMAC-SHA256) in the integrations table with a key held in TOKEN_ENCRYPTION_KEY. The app refuses to start in production if the key is missing.
  • Sensitive transcript text, onboarding-session state, and feedback-event payloads use the same Fernet wrapper via SQLAlchemy TypeDecorator.
  • Storage (Neon Postgres, AWS S3) applies provider-native encryption at rest; CallSight layers application-level encryption on top for the most sensitive columns.

Tenant isolation

  • Every tenant-scoped table carries a tenant_id foreign key. Queries are scoped in the application layer and enforced by PostgreSQL row-level security, which is configured via a per-session SET app.current_tenant_id.
  • API-key authentication looks up a SHA-256 hash; the plaintext key is never stored.
  • WebSocket /ws/live/{session_id} requires authentication before the upgrade handshake and verifies that the resolved tenant owns the session.
  • Cross-tenant aggregation is gated behind a per-tenant opt-in (features_enabled.data_use_for_improvement) enforced at the SQL layer.

Authentication

  • User sessions are managed by Clerk with MFA enforced on admin roles.
  • Machine-to-machine access uses scoped API keys with a configurable expiry. Key rotation is supported; keys are revocable from the admin UI.
  • OAuth callbacks validate a signed, time-limited state token; stale or tampered tokens are rejected at 400.

AI-specific safeguards

  • Presidio-based PII redaction is applied to caller transcripts before they reach Claude; the engine status is exposed via GET /api/v1/health/pii so degradation is observable.
  • Untrusted inputs fed to LLMs (caller speech, uploaded KB documents, inbound email bodies, CRM contact names, outcome notes) are wrapped in explicit data-delimiter blocks with an instruction to the model that content between them is data, not instructions.
  • LLM outputs are schema-validated before persistence; adversarial responses fall back to safe defaults.
  • Suggestions generated from third-party content (e.g., InferFromSources) are quarantined as TenantBriefSuggestion rows that require admin review — they never auto-write to the tenant brief.

Webhook integrity

  • Every outbound webhook is HMAC-SHA256 signed over a canonical timestamp + "\n" + body string. Subscribers should reject requests outside a ±300 s replay window.
  • Failed deliveries are persisted in webhook_deliveries and retried with exponential backoff (6 attempts, base 2). Operators can replay failed deliveries via the admin API.

Audit logging

An append-only audit_log table records authentication success, transcript reads, OAuth token use, API-key mint and revoke, webhook registration, and admin endpoint access. Database role permissions revoke UPDATE and DELETE on the table.

Secure development lifecycle

  • GitHub Actions runs detect-secrets on every push and PR. A scan hit fails the build.
  • Dependencies are pinned to minimum safe versions; critical libraries (cryptography, pypdf, python-docx, lxml) are bumped promptly after security releases.
  • The default CI workflow permission is contents: read; the legacy self-heal step that force-pushed LLM-generated fixes to main has been removed.

Reporting a vulnerability

Email security@callsight.ai. We will acknowledge within two business days and coordinate disclosure.
